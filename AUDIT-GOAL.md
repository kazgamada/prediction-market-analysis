# AUDIT 自律実行プロンプト（/goal 用）

> 使い方: 各ツールリポジトリの Claude Code で、以下の `/goal ...` ブロックをそのまま貼り付けて実行する。
> ルート直下の `AUDIT.md` を基準に、要件定義書の作成から main への push までを自律実行する。

---

```
/goal このリポジトリを「課金ユーザーに提供できる完成状態」へ到達させる。以下を自律的に最後まで実行し、各ステップで証拠（ファイルパス + 行番号 + コマンド出力）を残すこと。原因不明のまま修正せず、推測で実装しない。

1. 読込: ルートの AUDIT.md を全文読み、監査基準・共通UIデザイン規約・P0〜P3 の優先度定義を把握する。
2. 照合: 既存の要件定義書（docs/ 配下等。無ければ探索）と実装の現状を AUDIT.md の観点で突き合わせ、GAP を証拠付きで列挙する。
3. 要件定義書: このツール専用に最適化した要件定義書を docs/requirements.md に作成/更新する（AUDIT.md 共通規約 + このツール固有要件 + P0〜P3 と概算工数）。
4. 監査: AUDIT.md の手順で監査を実行。npm install → tsc --noEmit → npm run build → npm run lint を実際に走らせ、出力を証拠として記録し、P0/P1 を確定する。
5. コーディング: P0 → P1 の順に修正を実装する。各変更は最小単位、コミットは日本語の Conventional Commits。
6. 検査: 再度 tsc --noEmit → build → lint（テストがあれば実行）を通し、全て green を確認する。落ちたら原因を特定してから直す。
7. push: green を確認したら main へ直接 push する。

制約:
- 破壊的変更（DB drop、既存API削除、ファイル削除など）のみ事前に計画提示して承認を得る。それ以外は止まらず進める。
- 同じ箇所を3回直してダメなら仮説が誤り。リバートして最小再現に戻し、原因究明からやり直す。
- 応答・コミット本文・コードコメントは日本語。
```
