# DevOps デプロイチェックリスト

Vercel / GitHub Actions / Next.js スタック共通のデプロイ前確認と運用パターン。\
**新機能リリース・新環境立ち上げ・定期棚卸し** のいずれでも本 Skill を参照すること。

## 1. ブランチ戦略とコミット規約

```
main          ← 本番。直接 push 禁止。PR マージのみ
  └─ develop  ← 統合ブランチ（任意）
       └─ feature/<ticket>-<slug>
       └─ fix/<ticket>-<slug>
       └─ claude/<auto-generated-slug>   ← Claude Code セッション用
```

### 鉄則

- **最初に現在ブランチを確認**: `git branch --show-current`
- `main` への push は **明示の許可** があるときだけ
- タグ push が 403 になる場合（読み取り専用トークン）はローカル保持で OK
- コミットは **小さく・こまめに** push する（大きな WIP を溜めない）

## 2. CI パイプライン（GitHub Actions）

### 標準ワークフロー構成

```yaml
# .github/workflows/ci.yml
name: CI
on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck       # tsc --noEmit
      - run: npm run test            # vitest run
      - run: npm run build           # next build（型・バンドル最終確認）
```

### ポイント

| ステップ | コマンド例 | 目的 |
|---|---|---|
| Lint | `npm run lint` | ESLint / Biome |
| 型チェック | `npm run typecheck` | `tsc --noEmit` |
| テスト | `npm run test` | Vitest unit + integration |
| ビルド | `npm run build` | Next.js ビルド成功確認 |

## 3. 環境変数管理

### 3-1. 必須変数チェックリスト

```
# 認証・暗号化（最重要）
DATABASE_URL          — Supabase: Settings > Database > Connection string > URI
AUTH_SECRET           — openssl rand -base64 32
NEXTAUTH_SECRET       — 同上（Next-Auth v4 互換名）

# 外部 API
OPENAI_API_KEY        — platform.openai.com > API Keys
STRIPE_SECRET_KEY     — Stripe Dashboard > Developers > API keys

# Cron 保護
CRON_SECRET           — openssl rand -hex 32

# アプリ URL
NEXT_PUBLIC_APP_URL   — https://your-domain.vercel.app
```

### 3-2. 環境ごとの設定先

| 環境 | 設定方法 |
|---|---|
| ローカル開発 | `.env.local`（`.gitignore` 必須） |
| Vercel Preview | Vercel Dashboard > Settings > Environment Variables > Preview |
| Vercel 本番 | Vercel Dashboard > Settings > Environment Variables > Production |
| GitHub Actions | Repository > Settings > Secrets and variables > Actions |

### 3-3. 確認コマンド

```bash
# ローカルで未設定変数を検出
grep -r "process\.env\." src/ | grep -oP 'process\.env\.\w+' | sort -u

# Vercel CLI で現在の環境変数一覧
vercel env ls
```

> **棚卸し時**: 上記チェックリストと実際の設定を突き合わせ、不要変数は削除する。

## 4. API ルート保護

### 4-1. 認可ガードの実装パターン

```typescript
// lib/api-guard.ts
import { auth } from "@/auth";
import { NextRequest, NextResponse } from "next/server";

/** 一般認証ユーザー必須 */
export async function requireAuth() {
  const session = await auth();
  if (!session?.user) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
  return null; // OK
}

/** 管理者必須 */
export async function requireAdmin() {
  const session = await auth();
  if (session?.user?.role !== "admin") {
    return NextResponse.json({ error: "Forbidden" }, { status: 403 });
  }
  return null;
}

/** Cron ジョブ専用（CRON_SECRET 検証） */
export function requireCron(req: NextRequest): NextResponse | null {
  const secret = req.headers.get("authorization")?.replace("Bearer ", "");
  if (!process.env.CRON_SECRET || secret !== process.env.CRON_SECRET) {
    return NextResponse.json({ error: "Forbidden" }, { status: 403 });
  }
  return null;
}

/** 開発環境専用エンドポイント */
export function requireDev(): NextResponse | null {
  if (process.env.NODE_ENV === "production") {
    return NextResponse.json({ error: "Not Found" }, { status: 404 });
  }
  return null;
}
```

### 4-2. 新規 API ルートのガード確認

```bash
# 認可ガードが付いていないルートを検出
grep -rL "requireAuth\|requireAdmin\|requireCron\|requireDev\|verifySignature" \
  app/api/ --include="route.ts"
```

> ⚠️ 上記コマンドで出力されたファイルは **全て要確認**。意図的に公開する場合はコメントを残す。
