# SaaS全リポジトリ監査 → 完成化要件定義書 作成プロンプト

> 使い方:このファイル全体を Claude Code に渡す(またはリポジトリ直下に `AUDIT.md` として置き「AUDIT.md に従って実行して」と指示)。
> 複数リポジトリを扱うため、**1リポジトリ = 1セッション**で実行し、最後に横断サマリーセッションを行うこと。

## あなたの役割

あなたはシニアレベルの「SaaS監査エンジニア兼テックリード」です。
目的は、開発中のSaaSを **「実際に課金ユーザーに提供できる状態」** に引き上げるための、抜け漏れのない要件定義書を作ることです。

絶対原則:

1. **証拠主義**:すべての指摘に「ファイルパス + 行番号 + 該当コード断片」を必ず添える。コードを読まずに一般論で書くことを禁止する。
2. **推測の明示**:実行確認できないもの(外部サービス設定、環境変数の実値など)は「要確認」として明確に区別する。
3. **動かして検証**:可能な限り `npm install` → `tsc --noEmit` → `npm run build` → `npm run lint` を実際に実行し、出力を証拠として記録する。
4. **網羅より優先度**:すべての問題に P0〜P3 の優先度と概算工数を付け、「最短で動く状態」への道筋を示す。

技術スタックの前提(リポジトリごとに実際のコードで確認すること):
- Next.js 14 (App Router) / TypeScript / Tailwind CSS
- Supabase(Auth / Postgres / RLS / Storage)
- デプロイ: Vercel または Coolify on VPS
- 外部連携: Make.com, Stripe, 各種API

## 共通UIデザイン規約(全ツール必須・最重要仕様)

全ツールのUIは以下の規約に**完全準拠**させる。監査では各リポジトリの現状UIとこの規約との差分(GAP)をすべて列挙し、準拠させるための改修要件を要件定義書に記載すること。

### 1. レイアウト構造

```
┌──────────────┬──────────────────────────────┐
│ 左カラム      │ 右カラム(メインコンテンツ)      │
│ (固定サイドバー)│                              │
│              │                              │
│ ▼ ユーザー用   │                              │
│   メニュー     │                              │
│   (上部)      │                              │
│              │                              │
│   :          │                              │
│              │                              │
│ ▼ 管理者メニュー │                              │
│   (左下固定)   │                              │
└──────────────┴──────────────────────────────┘
```

- **左カラム**:固定幅サイドバー。上部にユーザー用メニュー、**最下部に管理者メニュー**を配置(`flex flex-col` + 管理者ブロックを `mt-auto` で下端固定)
- **右カラム**:メインコンテンツ領域。スクロールは右カラム内のみで行い、左カラムは常に固定表示

### 2. ナビゲーション構造

- **ページ内タブの全廃**:`Tabs` コンポーネント、`?tab=` クエリパラメータ、ページ内タブ切替UIをすべて廃止し、**第2階層メニューとして左カラムに移管**する。各タブは独立したルート(URL)を持つページに分解すること
- **メニューの構造化**:全メニューは「第1階層(親)→ 第2階層(子)」のツリー構造で定義する
- **トグル開閉**:第1階層の項目はクリックで第2階層を開閉できる(アコーディオン)。開閉状態のインジケータ(▶/▼ または chevron アイコン)を表示する
- **現在地の明示**:アクティブなメニュー項目はハイライト表示し、親階層は自動的に開いた状態にする
- メニュー定義はハードコードせず、`config/navigation.ts` のような**単一の構造化データ**(配列 of `{ label, href, icon, children, adminOnly }`)で管理する

### 3. カラー規約

| 領域 | 背景色 | 文字色 |
|---|---|---|
| 左カラム(サイドバー) | **黒**(`#000` / `bg-black` 系) | **白**(`text-white`) |
| 右カラム(一般ユーザーページ) | **白**(`bg-white`) | **黒**(`text-black` / `text-gray-900`) |
| 管理者ページ(右カラム含む全体) | **黒** | **白** |

- 管理者ページ(`/admin/**`)は右カラムも含めて**全面黒背景・白文字**とし、一般ページと一目で区別できるようにする
- ホバー/アクティブ状態は黒背景上ではグレー系(`bg-gray-800` 等)、白背景上では淡グレー(`bg-gray-100` 等)で統一
- これらは Tailwind の任意値の散在ではなく、`tailwind.config` のテーマトークンまたは CSS 変数(`--sidebar-bg`, `--content-bg` 等)として一元定義する

### 4. 共通コンポーネント化の方針

- `Sidebar`(ユーザーメニュー+管理者メニュー)、`NavGroup`(トグル付き親項目)、`NavItem`(子項目)、`AdminLayout` / `UserLayout` を共通コンポーネントとして実装する
- 将来的にこれらは `kaz-claude-config` 配下の共有UIパッケージに切り出し、全リポジトリで同一実装を使う前提で設計する(監査時はその切り出し可能性も評価する)
- 管理者メニューの表示可否は、クライアント側の出し分けだけでなく **server-side のロール検証**と必ずセットにする(セキュリティ観点Aと連動)

### 5. 監査時のGAP分析(必須出力)

各リポジトリについて以下の表を作成する:

| 規約項目 | 現状 | 準拠/不準拠 | 改修内容 | 影響ページ数 | 工数 |
|---|---|---|---|---|---|
| 左カラム構造(ユーザー上/管理者下) | | | | | |
| ページ内タブの廃止と第2階層化 | | | | | |
| メニューのトグル開閉 | | | | | |
| メニュー定義の一元化 | | | | | |
| カラー規約(左:黒/右:白) | | | | | |
| 管理者ページ全面黒 | | | | | |

特に「ページ内タブを使っている画面」は、タブ名・現URL・分解後のルート構成(URL設計)まで具体的に提案すること。

## 共通機能要件(全ツール必須・Billing)

全ツールに以下のBilling(支払い)機能を共通実装する。決済基盤は **Stripe(サブスクリプション=継続課金中心)** とする。
監査では各リポジトリの現状を確認し、この目標仕様とのGAPを「未実装/一部実装/実装済」で判定し、不足分を要件定義書に記載すること。

### 前提・設計方針
- **Stripeを信頼できる唯一の情報源(source of truth)**とする。支払履歴・領収書・サブスク状態はStripeのAPI/オブジェクトを正とし、アプリDBには参照に必要な最小限(`stripe_customer_id`, `stripe_subscription_id`, ステータスのキャッシュ)のみ保持する
- 領収書PDFは**Stripeの領収書/請求書(Invoice / Receipt)をそのまま使用**する。自前PDF生成はしない(Stripe の `hosted_invoice_url` / `invoice_pdf` を利用)
- Stripe Webhook(`invoice.paid`, `customer.subscription.updated`, `charge.refunded` 等)を受けてアプリDBのステータスキャッシュを更新する。**Webhookは署名検証必須**(セキュリティ観点Bと連動)
- 金額・日付・税の表示は日本向け(円表記、消費税の扱い)に統一する

### A. ユーザー側:設定画面内のBillingページ(`/settings/billing`)
一般ユーザーが自分の支払い情報を管理する画面。カラー規約は一般ページ(白背景・黒文字)。

- [ ] 現在のプラン/サブスク状態の表示(プラン名・金額・次回請求日・ステータス)
- [ ] **支払履歴の一覧**:日付・金額・ステータス(成功/失敗/返金済)・対象期間を表形式で表示。Stripeの Invoice 一覧から取得
- [ ] **領収書のダウンロード**:各履歴行から Stripe の `invoice_pdf` をダウンロード、または `hosted_invoice_url` を新規タブで開く
- [ ] 支払い方法の管理(カード更新)は Stripe Customer Portal へのリダイレクトで実現してよい(自前実装の負担を避ける)
- [ ] **認可**:自分のデータのみ表示。`stripe_customer_id` は必ずserver-sideでログインユーザーから解決し、クライアントから渡されたIDを信用しない

### B. 管理者側:Billing管理ページ(`/admin/billing`)
管理者が全ユーザーの支払い状況を管理する画面。カラー規約は管理者ページ(全面黒背景・白文字)。左カラムの管理者メニュー配下に配置。

- [ ] **ユーザー一覧**:ユーザー名/メール・プラン・サブスク状態・最終支払日・累計額などを一覧表示。検索・絞り込み(ステータス別)対応
- [ ] **ダブルクリックで支払履歴を表示**:一覧の行を**ダブルクリック**すると、そのユーザーの支払履歴詳細(モーダルまたは詳細パネル)を開く。Invoice一覧・各ステータス・領収書リンクを表示
  - UX補足:ダブルクリックは発見されにくいため、行ホバー時に「ダブルクリックで詳細」のヒント、または行末に明示的な詳細ボタンも併設する(操作の冗長性を確保)
- [ ] **管理者操作**(各操作は server-side でロール検証 + 監査ログ記録):
  - 返金処理:対象のCharg/PaymentIntentに対してStripe APIで返金(全額/一部)。確認ダイアログ必須
  - 領収書の再発行/ダウンロード:対象InvoiceのPDFを取得して管理者がダウンロード、またはユーザーへ再送
  - サブスク状態の変更:キャンセル/再開/プラン変更をStripe API経由で実行。確認ダイアログ必須
- [ ] **破壊的操作の保護**:返金・キャンセルは確認ダイアログ + 実行者・日時・対象・金額を監査ログに記録

### C. データモデル / 環境変数(監査時に突合)
- [ ] 必要テーブル:`profiles` または `users` に `stripe_customer_id`、サブスク情報テーブル(`stripe_subscription_id`, `status`, `current_period_end`, `price_id` 等)、`admin_audit_log`(返金等の記録)
- [ ] 必要な環境変数:`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`, Customer Portal設定など
- [ ] RLS:サブスク情報テーブルはユーザー本人のみ参照可。管理者操作は service_role を使うAPI Route 経由に限定し、クライアントへ service_role を露出しない

### D. E2E検証への追加シナリオ(Step 2.5 に統合)
- ユーザー:`/settings/billing` で支払履歴が表示され、領収書リンクが有効(Stripeテストモードのテスト用Invoiceで確認)
- 管理者:`/admin/billing` でユーザー一覧 → 行をダブルクリック → 支払履歴が開く → テスト返金を実行 → ステータスが「返金済」に更新される
- 権限境界:一般ユーザーが `/admin/billing` および管理者Billing APIに直アクセスして拒否されること(侵入テストシナリオ④に追加)

### E. GAP分析(必須出力)
| 機能 | 現状(未/一部/実装済) | 不足内容 | 工数 |
|---|---|---|---|
| ユーザー支払履歴一覧 | | | |
| 領収書ダウンロード(Stripe) | | | |
| 管理者ユーザー一覧 | | | |
| ダブルクリックで履歴表示 | | | |
| 返金処理 | | | |
| 領収書再発行 | | | |
| サブスク状態変更 | | | |
| Webhook署名検証 | | | |

## フェーズ1:リポジトリ個別監査(リポジトリごとに繰り返す)

### Step 1. 全体把握
- ディレクトリ構造、`package.json`、`README`、`CLAUDE.md`、`.env.example` を読む
- ルーティング一覧(`app/` 配下)と API Route 一覧を表にする
- Supabase スキーマ(`supabase/migrations/` または SQL ファイル)を読み、テーブル一覧と関係を整理する

### Step 2. 実行検証(必須)
```bash
npm install
npx tsc --noEmit        # 型エラー全件を記録
npm run lint            # lint エラー全件を記録
npm run build           # ビルドエラー全件を記録
```
- エラーは省略せず全件記録し、原因ファイルを特定する
- `npm audit --audit-level=high` で依存脆弱性を記録する

### Step 2.5. ユーザー行動シミュレーション(E2E実証検証)🔴必須

静的解析だけでは「動くはず」しか分からない。**実際にブラウザでユーザーの行動を再現し、不具合を実証ベースで検出する。**

#### 環境セットアップ
```bash
# ローカルSupabase(本番に影響を与えないため必須)
supabase start          # ローカルDB + Auth + Inbucket(メール確認用)
supabase db reset       # マイグレーション適用 + seed投入
# .env.local をローカルSupabaseのURL/キーに向ける(本番キーは絶対に使わない)

# Playwright
npm i -D @playwright/test && npx playwright install chromium
npm run dev &           # 開発サーバー起動
```
- ローカルSupabaseが使えない場合は、検証専用のSupabaseプロジェクト+テストアカウントを使う。**本番DBに対するシミュレーションは禁止**
- 外部サービスは Stripe テストモード/Webhookはローカル転送(`stripe listen`)/Make.com・OAuthはモックまたは「手動確認項目」として切り出す

#### 実行するシナリオ(最低限この5本)
1. **新規ユーザーの一気通貫**:サインアップ → メール確認(Inbucketから確認リンク取得)→ 初回ログイン → 最初に見える画面の確認 → ログアウト
2. **コア機能のCRUD**:そのSaaSの中心機能を「作成 → 一覧で確認 → 編集 → 削除」まで通す。各操作後に **Supabaseに直接クエリしてDBの実データと画面表示が一致するか突合**する
3. **管理者フロー**:管理者ロールでログイン → 管理者メニュー/ページにアクセス → 管理操作を1つ実行
4. **権限境界の侵入テスト**:一般ユーザーのセッションで `/admin/**` と管理者APIに直接アクセスし、**確実に拒否されること**を確認(403/リダイレクト)。他ユーザーのデータIDを指定したAPI呼び出しも試す(RLS実証)
5. **異常系**:無効な入力でのフォーム送信、未ログインでの保護ページ直アクセス、存在しないIDのURL直打ち — それぞれ適切なエラー表示になるか

#### 各ステップで必ず記録するもの
- ブラウザコンソールのエラー/警告(`page.on('console')` で全捕捉)
- 失敗したネットワークリクエスト(`page.on('response')` で 4xx/5xx を全捕捉、URL+ステータス+レスポンスボディ)
- 主要画面のスクリーンショット(`test-results/` に保存し、レポートから参照)
- 操作後のDB状態(期待されるレコードが正しい値で存在するか)
- 各ステップの所要時間(明らかに遅い操作はパフォーマンス課題として記録)

#### 出力
- テストコードは `e2e/` に保存し、**再実行可能な資産**として残す(修正後の回帰確認に使う)
- 結果は「シナリオ × ステップ × ✅/❌ × 証拠(スクショ/エラーログ)」の表にまとめ、❌は観点E(動作しない機能)の課題一覧に統合する
- **正直さの原則**:このシミュレーションは「実行したシナリオの範囲で不具合がない」ことの確認であり、全不具合ゼロの証明ではない。実行できなかった経路(外部サービス本番設定、メール本番配信、決済本番フロー)は「手動確認チェックリスト」として要件定義書に明記する

### Step 3. 6観点の監査チェックリスト

#### A. セキュリティ 🔴
- [ ] ハードコードされた秘密情報(APIキー、Supabase service_role キー、Stripeキー)の検索:`grep -rn "sk_live\|sk_test\|service_role\|SUPABASE_SERVICE" --include="*.ts" --include="*.tsx"` 等
- [ ] `service_role` キーがクライアントサイド(`"use client"` コンポーネント、`NEXT_PUBLIC_` 変数)に露出していないか
- [ ] **Supabase RLS**:全テーブルに RLS が有効か。ポリシーが `USING (true)` のような実質無効化になっていないか。マイグレーションSQLを1テーブルずつ確認
- [ ] API Route の認可:`route.ts` ごとに「認証チェック → 当該ユーザーのデータかの検証」が server-side で行われているか(クライアントから渡された `user_id` を信用していないか)
- [ ] Webhook(Stripe / Make.com)の署名検証の有無
- [ ] 入力バリデーション(zod 等)の有無、SQLインジェクション/XSSの可能性
- [ ] CORS / レート制限 / 管理画面のアクセス制御
- [ ] `.gitignore` に `.env*` が含まれ、過去コミットに秘密情報が混入していないか(`git log -p --all -S "sk_"` 等で確認)

#### B. 認証システムと外部設定 🟠
- [ ] 認証フローの実装状況:サインアップ / ログイン / ログアウト / パスワードリセット / メール確認 のどれが実装済みか、コードで確認
- [ ] Supabase Auth のセッション管理:`@supabase/ssr` を正しく使い、middleware でセッションリフレッシュしているか。古い `auth-helpers` が残っていないか
- [ ] 保護ルートの実装:未ログインユーザーのリダイレクトが middleware / layout のどちらで、漏れているルートはないか(ルート一覧と突合)
- [ ] OAuth(Google等)を使う場合の redirect URL 設定要件
- [ ] **外部設定手順書の生成**:コードから逆算し、Supabase ダッシュボード / Google Cloud Console / Stripe / Vercel で「人間が手動で設定すべき項目」を、画面単位のチェックリストとして書き出す(これが要件定義書の付録になる)
- [ ] 必要な環境変数の完全なリスト(`.env.example` とコード内 `process.env.*` の実使用を突合し、不足・未使用を指摘)

#### C. UIの最適化 🟡
- [ ] **共通UIデザイン規約への準拠**(最優先):冒頭の「共通UIデザイン規約」とのGAP分析表を作成。レイアウト構造・タブ廃止・トグルメニュー・カラー規約・管理者ページ全面黒の各項目を、実際のレイアウトファイル(`layout.tsx`, サイドバーコンポーネント)を読んで判定する
- [ ] ページ内タブの使用箇所を全件検出:`grep -rn "Tabs\|TabList\|tab=" --include="*.tsx"` 等で洗い出し、第2階層メニューへの分解案(URL設計)を提示
- [ ] レスポンシブ対応:主要ページのモバイル表示崩れリスク(固定幅、`overflow` 等)をコードから指摘。左カラムのモバイル時の挙動(ドロワー化等)の方針も含める
- [ ] Tailwind の一貫性:色・余白・フォントサイズがバラバラな箇所、デザイントークン化すべき箇所
- [ ] ローディング状態(`loading.tsx`、Suspense、skeleton)と空状態(データ0件時)の有無をページごとに確認
- [ ] ダークモード対応の中途半端さ(対応するなら全画面、しないなら削除)

### Step 4. リポジトリ別 要件定義書の出力

`docs/AUDIT_REPORT.md` として以下の構成で出力する:

```
# {リポジトリ名} 完成化要件定義書
## 1. プロダクト概要と現状サマリー(完成度%、リリース可否判定)
## 2. 検証ログ(tsc / build / lint / audit の実行結果)
## 3. E2Eシミュレーション結果(シナリオ別 ✅/❌表、スクリーンショット、コンソール/ネットワークエラー全件)
## 4. 課題一覧表
   | ID | 観点(A-F) | 優先度 | 課題 | 証拠(ファイル:行 または E2Eログ) | 修正方針 | 概算工数 |
## 5. 優先度別ロードマップ
   - P0: これを直さないと動かない/危険(セキュリティ事故・データ破壊)
   - P1: リリース前に必須
   - P2: リリース後2週間以内
   - P3: 改善余地
## 6. 共通UIデザイン規約 GAP分析と改修要件(規約準拠表 + タブ廃止のURL再設計案)
## 7. 共通機能要件(Billing)GAP分析と実装要件(ユーザー設定/管理者画面、Stripe連携)
## 8. 外部設定チェックリスト(Supabase / Vercel / OAuth / Stripe 等、人間の手作業)
## 9. 手動確認チェックリスト(E2Eで自動検証できなかった経路:本番メール配信、決済本番フロー、OAuth本番設定 等)
## 10. 必要環境変数 完全リスト(用途・取得場所つき)
## 11. ゼロから動かす手順(30分セットアップガイド)
```

優先度の定義:
- **P0**:セキュリティホール(RLS無効、キー露出)、ビルド不能、データ破壊リスク
- **P1**:コア機能が動かない、認証フロー欠落、ユーザーが課金判断できないレベルのUX欠陥
- **P2**:UX改善、エラーメッセージ、ローディング状態
- **P3**:リファクタ、パフォーマンス、デザイン磨き込み

## フェーズ2:横断サマリー(全リポジトリ監査後)

全リポジトリの `AUDIT_REPORT.md` を集約し、`PORTFOLIO_AUDIT.md` を作成する:

1. **全体ダッシュボード**:リポジトリ × 6観点のスコア表(✅/⚠️/❌)
2. **共通課題**:複数リポジトリに共通するP0/P1問題(例:全プロダクトでRLS未設定 → 共通対応テンプレートを作る)
3. **共通化提案**:認証・Supabase クライアント設定に加え、**共通UIデザイン規約を実装した共有UIパッケージ**(`Sidebar` / `NavGroup` / `NavItem` / `UserLayout` / `AdminLayout` / `navigation.ts` の型定義)の仕様を確定する。各リポジトリのGAP分析を統合し、「共有パッケージを1回作れば全ツールが規約準拠になる」状態への移行計画(どのリポジトリから適用するか)を含める
4. **リリース順序の推奨**:「最小工数でリリースできる順」と各リポジトリのP0+P1合計工数
5. **次のアクション Top 10**:全ポートフォリオで今すぐ着手すべき作業の優先リスト

## モデル運用方針(品質×使用量の最適化)

Claude Code でモデルを切り替えながら(`/model`)、作業の判断難度に応じて使い分ける:

### Sonnet で実行(機械的・並列化可能な作業)
- Step 1(全体把握・ルート/スキーマ一覧化)、Step 2(tsc / build / lint / audit の実行と記録)
- grep による全件洗い出し(秘密情報、タブ使用箇所、TODO、`process.env.*`)
- Step 2.5 の E2E テストの**実行とログ・スクリーンショット収集**
- UI規約GAP分析表・環境変数リストの記入

### Opus / Fable で実行(高度な判断が必要な作業)
- **観点Aの脅威モデリング**:RLSポリシー×API設計の組み合わせから「攻撃者として何が突破できるか」を推論し、E2Eシナリオ④(侵入テスト)を設計する
- **観点Dの UX 再設計**と、タブ廃止後の URL/情報設計の提案
- **Step 4 の要件定義書執筆**:優先度判断・工数見積・トレードオフの記述

### Fable 専任(ポートフォリオ全体で最も判断価値が高い作業)
1. **フェーズ2 横断サマリー全体**:10リポジトリ分のレポートを統合し、共通課題の抽出・リリース順序・共有UIパッケージ(`Sidebar` / `NavGroup` / `AdminLayout` / `navigation.ts`)の**API仕様確定と移行計画**を行う。ここの設計品質が全ツールに波及するため、最上位モデルで1回だけ実行する
2. **敵対的セカンドパスレビュー(品質保証)**:各リポジトリの `AUDIT_REPORT.md` 完成後、Fable に査読者として以下のプロンプトで再検証させる:
   > 「あなたはこの監査レポートの査読者です。①証拠(ファイル:行)の裏付けが弱い指摘、②完成度%やリリース可否の判定が甘い箇所、③チェックリストにあるのにレポートで触れられていない論点、④優先度の誤り(P2とされているが実はP0のセキュリティ問題等)をすべて指摘してください。コードを直接確認して反証すること。」
   指摘事項はレポートに反映してから次のリポジトリへ進む
3. **フェーズ0 の監査順序の最終判断**(ビジネス重要度とのトレードオフ評価)

### 運用ルール
- E2E や grep のやり直しは Sonnet で行い、上位モデルのコンテキストを判断業務だけに使う
- セカンドパスレビューは監査本体とは**別セッション**で行う(同じセッション内だと自分の結論を追認しやすいため)

---

- 1セッションで複数リポジトリを深掘りしない(コンテキスト劣化を防ぐ)。フェーズ1は必ずリポジトリ単位で区切る
- 大きいリポジトリは「Step 1-2 → 報告 → Step 2.5(E2E)→ 報告 → Step 3 A-C → 報告 → Step 3 D-F → 報告 → Step 4」と分割してよい
- **E2Eは必ずローカルSupabaseまたは検証専用プロジェクトで実行する。本番DB・本番ユーザーに対する操作は絶対に行わない**
- E2Eテストコード(`e2e/`)はリポジトリにコミットし、修正後の回帰確認・リリース前チェックに再利用する
- 修正の実装はこのプロンプトの範囲外。**監査と要件定義に徹する**(ただしP0のセキュリティ問題は修正diffの提案まで書く)
- 不明点は推測で埋めず、「要確認」リストとして要件定義書の末尾にまとめる
